Introduction

The Gateway Terminal to Host Guide is a comprehensive document designed to facilitate the integration of terminal devices with the Botim Money for Business (BMB) payment system. This guide is essential for merchants and developers who need to understand the detailed steps and API interactions required for seamless payment processing and device management.

The following chapters are covered in this document:

  • Protocol: Establishes the foundation for merchant-gateway connectivity through RESTful API specifications, envelope packaging standards, and signature mechanisms.
  • Device Registration Flow: Guides users through the merchant console login process, device information registration, and the review workflow.
  • POS Device Activation Flow: Covers device activation methods including QR code scanning and API-based activation.
  • POS Transaction Flow: Encompasses the complete payment lifecycle including Purchase, Reversal, and Refund flows.
  • Business API: Defines core entities and provides comprehensive API specifications for device activation, order management, and payment processing.
  • POS Error Codes: Catalogs error classifications with descriptions, root causes, and recommended troubleshooting approaches.
  • Receipt Format: Establishes mandatory and optional receipt formatting standards for POS transactions.
  • POS Terminal Application Specification: Defines the terminal application architecture with TPDU, Message Header, and ISO 8583 messaging.
  • Appendix: Specifies PIN block calculation, EMV tag requirements, and comprehensive response codes.

2.1 Common Layer

The common layer covers RESTful API details, envelope packaging, and signature mechanisms.

  • API URL: https://uat.test2pay.com/pos-gateway/
  • Signature key type: RSA-2048

2.1.1 HTTP Parameters

  • Method: POST
  • HTTP Headers: Content-Type: application/json • Accept: application/json • Charset: UTF-8 • Time Format: ISO8601

The outer envelope converts the secured content into a string and signs it. The recipient verifies the signature before decoding. The HTTP/HTTPS message body parameters:

FieldTypeMDescription

contentStringMThe content of the request.
signatureStringMAn alphanumeric string.
signatureVersionStringMThe version of signature (e.g., "1.0").

issuer

FieldTypeMDescription
issuerCodeStringMThe code of issuer. e.g., "PP352719030008899".
issuerTypeStringMThe type of issuer: POS_DEVICE or MEMBER.

2.2.2 Body

The request payload must be a hexadecimal-encoded string representing ISO 8583 data. Structure: TPDU + Message Header + Data Elements P0–P63.

POS Transaction Flow

5.1 Purchase

The Call Purchase API initiates bank card purchase transactions and retrieves the corresponding transaction results.

5.1.1 Transaction Processing Procedure

  1. Initiate Transaction — The cashier enters the transaction details into the POS terminal.
  2. Construct API Request — POS prepares a request to /terminal-to-host/8583 with ISO 8583 elements.
  3. Transmit to POS Gateway — Endpoint: https://uat.test2pay.com/pos-gateway/, Path: /terminal-to-host/8583
  4. Handle Response — Response contains ISO 8583 transaction elements.
  5. Post-Transaction — Print receipt on success; display error on failure; initiate reversal on exception.
ℹ Note
Body must be hexadecimal-encoded ISO 8583 data: TPDU + Message Header + Data Elements P0–P63. See Purchase API.

5.2 Purchase Reversal

Calls the Purchase Reversal API to reverse the original purchase.

5.2.1 Scenario 1: Automatic Reversal

This workflow handles automatic reversal when a timeout or response code W1 is received. Response codes 00, 12, and 25 all indicate a successful reversal.

5.2.2 Scenario 2: Manual Reversal

The cashier enters the trace number to locate the original purchase transaction. Transaction data must be stored in the terminal's local database — recommended retention period: 180 days.

ℹ Note
See Purchase Reversal API for details.

5.3 Refund

Calls the Refund API to perform refund operations.

5.3.1 Refund Transaction Procedure

  1. Initiate — Enter Amount, Original transaction date, and POS center unique identifier (field 60.4).
  2. Construct Request — ISO 8583 elements for /terminal-to-host/8583.
  3. Transmit — Endpoint: https://uat.test2pay.com/pos-gateway/
  4. Handle Response — Process ISO 8583 response elements.
  5. Post-Transaction — Print receipt on success; show error on failure.
ℹ Note
See Refund API for details.

POS Terminal Application Specification

9.1 Message Structure

Messages sent from the POS terminal to the host consist of three components: TPDU (Transport Protocol Data Unit), Message Header, and Application Data (ISO 8583 Message).

9.1.1 Message Layout

TPDU: ID: 60H • Destination Address: NN NN • Source Address: NN NN (10 bytes, compressed to 5 using BCD)

Message Header: Application Category: 61H • Software Version: 01H • Terminal Status: 0H • Processing Requirement: N1 • Software Sub-version: N6 (12 bytes, compressed to 6 using BCD)

Application Data: ISO 8583 Message — Transaction Data of Variable Length

9.1.2 Processing Requirements

CodeDescription
0No processing required
1Re-sign-in
2Notify terminal to update public key
3Download IC card parameters
4Download contactless business parameters

9.2 Data Type Definitions

9.2.1 Data Element Types

CodeDescription
AAlphabetic, left-aligned, right-padded with spaces
ANAlphanumeric, left-aligned, right-padded
ANSAlphanumeric + special characters
BBinary bits
NNumeric, right-aligned, zero-padded; last two digits = cents
LL2-digit length prefix (variable field)
LLL3-digit length prefix (variable field)

9.2.2 Symbol Definitions

SymbolDescription
MMandatory — Field must appear in the message.
CConditional — Field appears only under certain conditions.
OOptional — May be included at sender's discretion.

9.3 Field Specifications

The terminal application uses ISO 8583 message fields with specific encoding requirements. Key field types include BCD compression for numeric fields, BINARY for PIN and bitmap fields, and ASCII for text identifiers. DUKPT key management is used for PIN block encryption per ISO 9564-1 Format 0.

Appendix: PIN Block

10.1 PIN Block Calculation

PIN block calculation uses ISO 9564-1 Format 0 with 3DES encryption. The PIN block is formed by XOR-ing the PIN data block with the PAN data block, then encrypting with the terminal's DUKPT-derived key.
